# C2 Infrastructure Guide

Transport architecture, mTLS, malleable profiles, redirector, and domain fronting.

---

## Transport Stack

Fitnah runs three parallel transport channels. All three are active once established — they do not replace each other, they coexist.

```
Priority 0  Telegram     api.telegram.org:443         Always active
Priority 1  Discord      discord.com:443              Activates on Telegram failure
Priority 2  Turnt        *.relay.teams.microsoft.com  Activates on demand (stealth)
```

The router sends every outbound message on Telegram (priority 0). If three consecutive Telegram sends fail, the router transparently switches to Discord. When Telegram becomes reachable again, the router switches back automatically.

Turnt is always a parallel channel — it doesn't replace Telegram or Discord. Once the tunnel is up, the implant simultaneously beacons via Telegram and maintains the TURN data channel.

### Failover threshold

```yaml
c2:
  failover_threshold: 3   # consecutive failures before switching
```

## Telegram Transport

The primary C2 bus. Each implant gets its own private Telegram group (`Session.group_id`). The bot sends `{"type":"TASK",...}` JSON and receives `{"type":"ACK",...}` or `{"type":"CHECKIN",...}`.

The bot **never** builds payloads or executes code on its own. It is purely a message relay.

### Configuration

```yaml
telegram:
  token: "BOT_TOKEN"         # @BotFather token
  operator_chat_id: 123456   # your personal Telegram user ID
```

### Per-agent groups

Each session has a dedicated group so multiple implants don't see each other's traffic. When a new implant checks in, the kernel creates a new group and records `Session.group_id`.

---

## Discord Transport (Fallback)

Activates automatically when Telegram fails. Uses a Discord bot + a dedicated text channel. All the same JSON protocol, different transport.

### Configuration

```yaml
discord:
  token: "DISCORD_BOT_TOKEN"
  operator_channel_id: 987654321
  enabled: true
```

Set `enabled: false` to disable Discord entirely (single-channel mode).

## HTTP Listener

A lightweight aiohttp server that accepts implant HTTP beacons. Supports:
- AES-256-GCM encrypted bodies (`X-Encrypted: 1` header)
- Optional TLS (self-signed or operator-provided cert)
- mTLS (per-agent client certs, signed by Fitnah CA)
- Malleable C2 profiles (custom URIs, headers, body prepend/append)

### Configuration

```yaml
http:
  enabled: true
  host: "0.0.0.0"
  port: 8888
  agent_key: "change-this-secret"
```

### Endpoints

```
POST /checkin   Implant sends CHECKIN JSON, receives pending tasks
POST /ack       Implant sends task result ACK
```

Both URIs are overridden by the active C2 profile.

### Request authentication

Every request must include:
```
X-Agent-Key: <agent_key from config>
X-Agent-Id: <agent_id>
```

Missing or incorrect `X-Agent-Key` returns HTTP 403. With mTLS enabled, the client cert is also verified against the CA.

### Encrypted body protocol

```
Request body:   base64(AES-GCM-encrypt(JSON payload))
Request header: X-Encrypted: 1
Response body:  base64(AES-GCM-encrypt(JSON response))
Response header: X-Encrypted: 1
```

The AES key is derived from `agent_key` in the config.

## TLS

### Auto self-signed cert

```yaml
http:
  auto_tls: true
```

Generates `data/tls/server.crt` and `data/tls/server.key` on first start using RSA-2048. The implant must trust this cert (set `verify_tls: false` in implant config for self-signed).

### Operator-provided cert

```yaml
http:
  tls_cert: "/path/to/server.crt"
  tls_key: "/path/to/server.key"
```

Use a Let's Encrypt cert for a domain you control — implants verify normally.

## mTLS — Per-Agent Certificate Authority

Ported from [BishopFox/Sliver](https://github.com/BishopFox/Sliver) `server/certs/`. Each agent gets a unique leaf cert signed by a Fitnah-controlled CA.

### Design

```
Fitnah CA (RSA-4096, 10 years)
    └── Agent abc123 leaf cert (RSA-2048, 1 year)
    └── Agent def456 leaf cert (RSA-2048, 1 year)
    └── Agent ghi789 leaf cert (RSA-2048, 1 year)
```

- CA is generated once on first use, saved to `data/tls/ca.crt` and `data/tls/ca.key`
- Each leaf cert has `CN=<agent_id>`, `SAN=<agent_id>`
- The listener requires `CERT_REQUIRED` — any connection without a valid leaf cert is rejected at the TLS layer, before any HTTP code runs
- Revoking adds `agent_id` to `data/tls/revoked.json` — the listener rejects the agent in `_auth_ok()` before processing its request

### Enable mTLS on listener

```yaml
http:
  mtls: true
  tls_cert: "data/tls/server.crt"
  tls_key: "data/tls/server.key"
```

### Burn an agent

```
# In REPL:
burn abc123
```

This revokes the agent's leaf cert. The agent's next beacon is rejected at the TLS layer. Other agents continue unaffected.

### Verify CA

```bash
openssl x509 -in data/tls/ca.crt -noout -text
openssl verify -CAfile data/tls/ca.crt data/tls/agents/abc123/client.crt
```

## Malleable C2 Profiles

Profiles disguise beacon traffic as legitimate application traffic. Configured in `fitnah/c2/profiles.py`.

### Built-in profiles

| Name | Mimics |
|---|---|
| `default` | Plain JSON |
| `jquery` | jQuery CDN update requests |
| `office` | Microsoft Office telemetry |
| `windows_update` | Windows Update API |

### Apply a profile

```
# In the REPL, when setting up the HTTP listener:
listener start --profile jquery

# Or in builder:
builder -f https-ps1 -a abc123 --profile jquery
```

### What a profile controls

```python
@dataclass
class C2Profile:
    checkin_uri:    str   = "/jquery-3.6.0.min.js"
    ack_uri:        str   = "/jquery-3.6.0.min.js"
    headers:        dict  = {"Content-Type": "application/javascript"}
    body_prepend:   bytes = b"/*! jQuery v3.6.0 */"
    body_append:    bytes = b""
    user_agent:     str   = "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
```

The listener strips `body_prepend` and `body_append` before decrypting. The implant adds them before sending.

### Custom profile

```python
# fitnah/c2/profiles.py
PROFILES["my_profile"] = C2Profile(
    checkin_uri  = "/api/v2/metrics",
    ack_uri      = "/api/v2/metrics",
    headers      = {"Content-Type": "application/json", "X-Api-Version": "2"},
    body_prepend = b'{"data":',
    body_append  = b"}",
)
```

## 3-Tier Redirector

Protects the real C2 listener from exposure. Traffic flows:

```
Internet → Edge proxy (nginx, looks like WordPress)
                ↓   X-Sync-Token header present
           Sinkhole (filters scanners, wrong geos, wrong ASNs)
                ↓   passes all filters
           Real C2 listener (never exposed directly)
```

### Filters applied by the sinkhole

| Filter | Default |
|---|---|
| Geofencing | Only allow configured countries |
| ASN blocklist | Block Google, Amazon, DigitalOcean, Shodan, etc. |
| Header check | Require custom `X-Sync-Token` header with correct value |
| Rate limiting | 10 requests/minute per IP |
| UA check | Reject common scanner/tool UAs |

### Configure the redirector

```yaml
redirector:
  edge_host: "0.0.0.0"
  edge_port: 80
  real_c2_host: "127.0.0.1"
  real_c2_port: 8888
  sync_token: "change-this-token-to-something-long"
  allowed_countries: ["US", "GB", "DE", "CA"]
  decoy_response: "decoy/wordpress_home.html"
```

### Start the redirector

```
# In REPL:
redirector start
```

## Domain Fronting

Routes beacon traffic through CDN providers to hide the real C2 server.

```
Implant SNI → cdn.cloudflare.com
Implant Host header → your-c2.yourdomain.com
```

CDN forwards based on `Host` header → real C2. Network logs show only `cdn.cloudflare.com`.

### Configure domain fronting

```yaml
domain_fronting:
  provider: "cloudflare"       # cloudflare | fastly | azure
  front_domain: "cdn.cloudflare.com"
  real_domain: "c2.yourdomain.com"
  enabled: true
```

## Network Diagram

```
Target host
  │
  │  Telegram / Discord (HTTPS, port 443)
  │  HTTP beacon (port 8888, or 443 with TLS)
  │
  ↓
Corporate proxy / firewall
  │
  │  Telegram: allowed (universal whitelist)
  │  Discord: usually allowed
  │  Turnt (Teams relay): universally allowed (Teams is business-critical)
  │  HTTP: blocked by strict proxies → use HTTPS with profile
  │
  ↓
Internet
  │
  ├── api.telegram.org:443  ──────────────────────► Telegram servers → your bot
  ├── discord.com:443 ─────────────────────────────► Discord servers → your bot
  ├── *.relay.teams.microsoft.com:443  ───────────► Teams TURN servers → turnt tunnel
  └── your-redirector.com:443 ───────────────────► 3-tier redirector → real C2
```

## OPSEC Notes

- **Never expose the real C2 listener directly** — always put the 3-tier redirector in front
- **Use HTTPS** — plain HTTP beacon traffic is trivially detected by DPI
- **Use mTLS** — prevents any non-implant client from interacting with your C2
- **Rotate profiles** — use different profiles across engagements
- **Separate bots per engagement** — use fresh Telegram/Discord bots, never reuse tokens
- **Burn compromised agents immediately** — `burn <agent_id>` revokes the cert; the agent cannot reconnect even if it still has the token
